◉ EP1 开篇03 / 26

三个正确 —— 不会过时的底层逻辑

① 做正确的事

先把需求搞清楚。高效地做错事——是最可怕的。

⚠ 失败模式：需求没理清就让AI写代码

② 正确地做事

用AI当杠杆，但要用对地方。方案对比→任务拆解→风险提示→你拍板。

⚠ 失败模式：把AI当搜索框

③ 次次做对

靠机制保证：回滚点 + 测试验证 + 文档沉淀。

⚠ 失败模式：只追求速度，跳过验证

💡 核心洞察：答案变得廉价，好的问题变得昂贵。AI让获取答案成本趋零，提出正确问题的能力反而成了稀缺资源。

2.1 四步提问法 —— 好问题决定好答案

① 背景→ ② 目标→ ③ 假设约束→ ④ 验收标准

❌ 模糊提问

"帮我写一个排序函数"

✅ 精准提问

2.5 Git三原则 —— 给AI时代上保险

原则一：频繁提交

每完成一个小功能点就 commit。AI跑偏了可以随时回滚，损失最小化。

原则二：意图清晰

commit message写清"为什么改"。AI生成的代码更需要人类标注意图。

原则三：AI操作前先commit

让AI做大改动之前，先保存现场。
git diff 审查变更。

💡 进阶技巧：git worktree —— 同一仓库开多个工作目录，AI在分支上折腾不影响主线。

2.6 安全并行 —— 从线性到并行的效率跃迁

核心原则：工作集正交

同一时刻，同一文件只归一个执行者。
多个AI窗口并行时，各自操作不同文件集。

7种AI应用场景（不只写代码）

代码编写代码审查测试生成文档撰写数据分析方案对比内容创作

线性 vs 并行

线性推进并行探索
速度慢，串行等待快，同时进行
风险单点失败全停互不影响
适用强依赖链任务可拆分子任务

2.10 工具选型 —— 工具层 vs 模型层

层级定义类比典型
工具层在哪干活车间VS Code+Copilot, Cursor, Windsurf
模型层装什么引擎发动机GPT-4.1, Claude Sonnet 4, Gemini
💡 两者独立选择，不要混为一谈。工具层决定交互体验，模型层决定推理质量。

三层决策树

第一层：先把路选对 → 选工具层
第二层：再把车配够 → 选模型层
第三层：按路况换挡 → 按任务动态切换

建议"主力+备用"组合，不深度绑定单一供应商

2.11 提示词协议化 —— 系统级指令管理

三层管理体系

层级文件作用域
全局copilot-instructions.md整个工作区
专项*.instructions.md按类型/场景
任务级*.agent.md特定Agent

上层定义长期稳定约束，下层定义局部差异

四要素 & 七反模式

好指令四要素：
上下文 + 角色 + 任务 + 输出格式

避坑：不写敏感信息、不堆废话、不留过期规则、不写冲突规则、不缺验收口径、不工具绑定过死、不过度细节堆砌

三层提示词管理体系 — 可视化

作用域：整个工作区

🌐 全局层

.github/copilot-instructions.md

代码风格 · 项目结构 · 安全约束 · 通用规范

作用域：按类型/场景

📂 专项层

*.instructions.md

Python规范 · 安全指南 · 无障碍 · 性能优化

作用域：特定Agent或任务

🎯 任务级

*.agent.md / *.prompt.md

博客写作Agent · 代码重构Agent · 调研Agent

设计原则

1稳定性递增：全局层最稳定，任务级最灵活
2继承覆盖：下层可以覆盖上层的局部规则
3单一职责：每个文件只管一类约束
4可审计：所有指令纳入版本管理
就像软件工程中的配置分层——全局 > 模块 > 环境，提示词也需要结构化治理而非零散堆叠。

九条红线 — 风险等级与应对措施

#红线风险典型错误应对措施
❶不要只看速度高用"生成行数"衡量产出，忽略正确性建立验收标准，速度只是附带收益
❷不要跳过验证高AI输出直接合并，不做代码审查每次AI输出必须过人类审查再合入
❸不要只对话不沉淀中每次从零对话，经验无法复用对话有价值就提炼为文档或指令
❹不要工具先行中先买工具再想需求，为用而用先定义问题，再选适合的工具
❺不要过度依赖高离了AI就不会写代码/做分析保持核心技能，AI是放大器非替代品
❻警惕AI幻觉高AI编造不存在的API/数据关键事实必须二次验证
❼警惕幸存者偏差中只看成功案例，忽视失败场景主动收集反面案例，建立失败库
❽经验≠永恒结论低过去有效的策略照搬到新场景定期回顾，模型/工具更新后重新评估
❾最小权限高给AI开放全部文件/网络/API权限权限按需开放，敏感操作人工确认

约束工程 · Constraint Engineering

核心公式

可用性 = f(模型能力) × g(约束质量)

模型能力你无法控制 · 约束质量你完全可以控制

层级控制方典型动作
希望层服务商模型升级、训练优化
保证层使用者写约束、定流程、做验证

约束质量五维度

完整性 — 覆盖了关键边界吗？
明确性 — 无歧义吗？
一致性 — 多条约束之间矛盾吗？
可验证性 — 对应可检查的输出吗？
可维护性 — 便于更新迭代吗？

从 Prompt Engineering 到 Harness Engineering

Vibe Coding

让AI凭感觉写
碎片代码，不可维护

Prompt Engineering

优化单次提示词
高质量单次输出

Spec Coding

🌟 Harness Eng.

系统级约束+流程+验证

Harness = "驾驭" —— 不是控制AI做什么，而是建立框架让AI在对的范围内发挥最大能力。

验证层：辨别AI输出质量
