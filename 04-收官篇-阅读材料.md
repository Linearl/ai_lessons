## 一、自动化工作流·从手动到半自动（对应2.9）

### 1.1 工作流四步搭建法

| 步骤 | 动作 | 关键检查项 |
|------|------|-----------|
| 1. 识别 | 列出每周高频重复任务清单 | 是否至少重复3次 |
| 2. 拆解 | 将任务拆成"人做"和"AI做"两部分 | 边界是否清晰 |
| 3. 串联 | 用脚本/Skill/MCP串联各步骤 | 步骤之间的输入输出是否对齐 |
| 4. 验证 | 跑通端到端流程并检查输出 | 是否有人工审核节点 |

### 1.2 工作流的组成要素

工作流本质上是"2.5版本管理 + 2.7重复封装 + 2.8文档体系"的组合应用：

- **版本管理**保证可回退
- **封装**把可重复步骤变成可调用的工具
- **文档体系**把工作流的输入/输出/规则显式化

### 1.3 典型工作流案例

- **分析报告流水线**：数据采集 → 清洗 → 分析 → 可视化 → 报告生成 → 人工审核
- **代码审查辅助**：PR提交 → AI初审 → 生成Review意见 → 人工复核 → 合并或打回
- **内容创作流水线**：素材收集 → 初稿生成 → 多轮对话优化 → 格式化输出 → 发布

## 二、工具选型·从迷茫到清晰（对应2.10·增强版）

### 2.1 核心概念：工具层 vs 模型层

很多人把"选AI工具"等同于"选模型"，这会导致选型混乱。正确的认知框架：

| 层级 | 定义 | 类比 | 典型选择 |
|------|------|------|----------|
| **工具层** | 你在哪个"车间"干活 | 车间 / 生产线 | VS Code + Copilot、Cursor、Windsurf |
| **模型层** | 车间里装的"发动机" | 发动机 | GPT-4.1、Claude Sonnet 4、Gemini 2.5 Pro |

> 工具层决定交互体验和工作流集成度；模型层决定推理质量和上下文处理能力。两者独立选择，不要混为一谈。

### 2.2 三原则选型法

| 原则 | 核心问题 | 落地做法 |
|------|----------|----------|
| **价格** | 月成本能否覆盖 | 先锁定次顶级订阅，不追最贵 |
| **效果** | 能否解决核心任务 | 拿真实任务做A/B测试，不看评测榜单 |
| **稳定性** | 生产环境可靠吗 | 观察连续7天的一致性，而非单次惊艳表现 |

### 2.3 次顶配订阅对比（截至2025年中）

| 服务 | 月费 | 核心能力 | 适合场景 |
|------|------|----------|----------|
| GitHub Copilot Pro | $10 | VS Code深度集成、Agent模式 | 已有VS Code习惯 |
| Cursor Pro | $20 | 代码专项优化、多模型切换 | 追求极致代码体验 |
| Claude Pro | $20 | 长上下文、严谨推理 | 文档分析、复杂推理 |
| ChatGPT Plus | $20 | 多模态、插件生态 | 综合日常使用 |
| Google AI Pro | $19.99 | 超长上下文（100万token）| 巨量文档处理 |
| Windsurf | $15 | IDE集成、实时协作 | 轻量级AI辅助 |
| DeepSeek | ¥0 | 开源、API灵活 | 预算敏感/自部署 |
| 通义/豆包 | ¥0–¥30 | 中文优化 | 国内网络/中文场景 |

> 表格反映发布时点信息，具体以各平台最新定价为准。

### 2.4 三层决策树

做选型时按这三层顺序决策：

```
第一层：先把路选对 → 选工具层（IDE/平台）
         ↓
第二层：再把车配够 → 选模型层（订阅/API）
         ↓
第三层：按路况换挡 → 按任务动态切换模型
```

**第一层**：如果团队已有VS Code生态和GitHub工作流 → Copilot是阻力最小的选择。如果团队追求代码编辑极致体验 → 考虑Cursor。

**第二层**：不要只订一个服务。建议"主力+备用"组合，例如Copilot（日常）+ Claude Pro（深度分析）。

**第三层**：同一工具层内可以切换模型。例如VS Code里可以选GPT-4.1、Claude Sonnet 4、Gemini等。

### 2.5 风险约束（与三层决策树并行）

| 约束项 | 要求 |
|--------|------|
| 数据合规 | 确认服务商的数据使用政策，敏感数据不上传 |
| 网络稳定 | 国内使用需确认可用性和延迟 |
| 可替换性 | 不深度绑定单一供应商 |
| 成本预算 | 团队人均月成本控制在合理范围 |

### 2.6 角色分层建议

| 角色 | 推荐方案 |
|------|----------|
| 个人开发者 | GitHub Copilot Pro（$10）作为起步 |
| 专业用户 | Copilot Pro + Claude Pro 双订阅 |
| 团队/管理者 | 统一工具层 + 按需开放模型层权限 |

## 三、提示词协议化·系统级指令管理（对应2.11）

### 3.1 从零散提示词到协议化管理

大多数人的提示词停留在"临时写、用完忘"阶段，这意味着每次都在重复造轮子。协议化的核心是让提示词变成可版本管理、可复用、可审查的工程资产。

### 3.2 典型提示词结构（最小可用版本）

```markdown
---
description: "一句话说明这条指令的用途"
applyTo: "适用的文件范围，如 **/*.py"
---

# 指令标题

## 核心规则
1. ...
2. ...
3. ...

## 示例
[好的例子]
[坏的例子]
```

### 3.3 三层管理体系

| 层级 | 文件 | 作用域 | 典型内容 |
|------|------|--------|----------|
| 全局 | copilot-instructions.md | 整个工作区 | 项目概况、通用编码规范 |
| 专项 | *.instructions.md | 按文件类型或场景 | Python规范、安全规则、代码审查标准 |
| 任务级 | *.agent.md / *.prompt.md | 特定Agent或任务 | 博客写作Agent、代码重构Agent |

### 3.4 四要素检查清单

一条好的提示词指令应包含：

| 要素 | 说明 | 举例 |
|------|------|------|
| **上下文** | 项目背景和约束 | "Python 3.10+，使用pathlib" |
| **角色** | AI应扮演的角色 | "你是安全审计专家" |
| **任务** | 具体要做什么 | "审查此PR中的SQL注入风险" |
| **输出格式** | 期望的输出形式 | "以Markdown表格输出，包含风险等级" |

### 3.5 七个反模式（管理维度）

> 博客正文列出的七个反模式侧重**内容质量**（不写敏感信息、不堆废话、不留过期规则等）。下表从**管理实践**角度做补充，两组反模式配合使用效果最佳。

| # | 反模式 | 后果 | 正确做法 |
|---|--------|------|----------|
| 1 | 一条指令塞太多规则 | AI执行飘忽 | 按场景拆分为多条专项指令 |
| 2 | 只说"不要做什么" | AI不知道该做什么 | 用正面描述替代否定句 |
| 3 | 没有示例 | AI理解偏差大 | 至少给一个好例子一个坏例子 |
| 4 | 没有版本管理 | 改了不知道改了啥 | 用Git管理，每次改动写commit message |
| 5 | 从不更新 | 规则与实际脱节 | 定期回顾，删除过时规则 |
| 6 | 只有个人知道 | 团队无法复用 | 放进项目仓库共享 |
| 7 | 规则相互矛盾 | AI行为不一致 | 全局-专项-任务三层优先级明确 |

## 四、安全红线·不可逾越的九条底线（对应2.12）

### 4.1 九条安全红线

| # | 红线 | 典型违规场景 | 防护措施 |
|---|------|-------------|----------|
| 1 | 不上传敏感数据 | 把客户数据粘贴给ChatGPT | 使用脱敏数据或本地模型 |
| 2 | 不盲信AI输出 | AI生成的SQL直接执行 | 所有AI输出必须人工审核 |
| 3 | 不跳过代码审查 | AI写的代码直接合并 | 遵循正常PR审查流程 |
| 4 | 不忽略许可证 | AI建议的代码可能有版权 | 检查代码来源和许可证 |
| 5 | 不泄露API密钥 | 在对话中粘贴密钥 | 使用环境变量或密钥管理服务 |
| 6 | 不让AI做最终决策 | 让AI决定上线时间 | AI给建议，人做决定 |
| 7 | 不省略测试 | "AI说没问题就行" | AI生成的代码同样需要测试 |
| 8 | 不过度依赖单一工具 | 全部工作绑定一个AI服务 | 保持可替代性 |
| 9 | 不忽视团队规范 | 个人用法覆盖团队标准 | 以团队规范为准 |

### 4.2 安全意识自检

- [ ] 我是否确认了AI服务的数据使用政策？
- [ ] 我是否在sensitive数据上使用了脱敏处理？
- [ ] 我是否对AI的输出做了人工审核？
- [ ] 我是否遵循了团队的代码审查流程？
- [ ] 我是否检查了AI建议代码的版权合规性？
