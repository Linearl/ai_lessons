## 一、Git版本管理·完整操作指南（对应2.5）

### 1.1 三条核心原则

1. **所有项目必须使用Git进行版本管理**，无论规模大小
2. **频繁提交、小步快跑**：每完成一个有意义的阶段就commit，不要等"全部做完"
3. **commit message写清意图**：方便回溯时快速定位到正确的版本

### 1.2 冷备份策略

| 策略 | 做法 | 场景 |
|------|------|------|
| 远端推送 | 及时push到GitHub/GitLab | 避免本地磁盘故障丢失 |
| 关键节点打Tag | 里程碑版本、上线前、大规模重构前 | 留快照便于回退 |
| 非代码资产冷备 | 分析报告、配置文件定期复制到独立目录或云盘 | 保护非Git管理的重要文件 |

### 1.3 AI操作的风险防护

- **大规模AI操作前先commit**：批量重构、文件重组等高风险操作前，确保当前状态已提交
- **善用git diff审查AI改动**：AI完成操作后，先用`git diff`审查变更范围和内容，确认无误再提交
- **必要时用分支隔离实验**：不确定效果的AI操作在独立分支上进行

### 1.4 git worktree对比历史版本

```bash
# 将历史分支/tag签出到临时目录
git worktree add ../temp-old-version v1.0.0

# 对比完成后及时清理
git worktree remove ../temp-old-version
```

> 建议将临时目录路径加入`.gitignore`，用完后及时清理worktree。

## 二、安全并行·多窗口与并行探索（对应2.6）

### 2.1 AI不只是写代码的工具

| 场景 | 典型任务举例 |
|------|------------|
| 数据分析 | 行业数据整理、趋势分析、报告生成 |
| 行业研究 | 产业链梳理、竞争格局分析、投资标的筛选 |
| 文档处理 | 文件整理归类、格式转换、内容提取与摘要 |
| 方案规划 | 旅游计划、活动策划、行李清单等结构化输出 |
| 问题分析 | Bug分析报告、故障堆栈解读、根因定位 |
| 项目分析 | 开源项目架构拆解、技术选型对比 |
| 知识整理 | 分享材料撰写、调研报告、内容精简 |

### 2.2 并行探索 vs 线性推进

| 模式 | 线性推进 | 并行探索 |
|------|----------|----------|
| 方案验证速度 | 串行耗时长 | 并行对比快 |
| 认知偏差风险 | 容易被首个方案锚定 | 打破单一路径依赖 |
| 决策质量 | 依赖单次判断 | 基于多方案证据做选择 |

适合并行探索的典型场景：
- 需求不明确：同时让AI输出"保守版/平衡版/激进版"
- 技术选型争议：并行比较实现复杂度、性能、维护成本
- 疑难Bug排查：并行从"并发/配置/数据"等不同假设路径定位

> 并行探索结束后必须收敛，明确选择标准并沉淀取舍理由。

### 2.3 按职责拆窗口

| 窗口 | 职责 | 典型任务 |
|------|------|----------|
| 窗口A | 分析与文档 | 代码理解、问题定位、报告沉淀 |
| 窗口B | 需求与方案 | 澄清需求、整理方案、定义验收 |
| 窗口C | 开发与测试 | 新功能实现、测试验证、问题修复 |

利用AI推理的等待窗口完成人工审查与验证工作。

### 2.4 工作集正交原则

**关键规则：每个窗口/Agent操作的文件集合不重叠。**

- 按模块或文件粒度划分各窗口的职责范围
- 同一时刻只有一个Agent修改特定文件
- 如果必须涉及同一文件，先完成一个窗口的任务并提交，再在另一个窗口继续

## 三、重复封装·从手动到自动化（对应2.7）

### 3.1 适用场景判断

当某类任务满足以下条件时，就不该每次手工提示：
- **高频重复**：每周至少做3次以上
- **步骤固定**：流程基本一致
- **依赖稳定**：输入输出格式不常变化

### 3.2 封装层级

| 封装方式 | 适用场景 | 复杂度 |
|----------|----------|--------|
| 提示词模板 | 高频对话模式 | 低 |
| 脚本 | 文件处理、数据转换 | 中 |
| Skill | 多步骤AI协作流程 | 中高 |
| MCP工具 | 需要外部API集成 | 高 |

### 3.3 快速封装技巧

不需要一开始就"设计"一个完美的工具。实际做法：

1. **先手动操作跑通一遍**
2. **直接让AI根据你刚才的操作过程进行总结和封装**
3. **检查封装结果、迭代优化**

从"手动做完"到"封装成工具"的成本，比你预期的低得多。

### 3.4 官方技能库推荐

- **GitHub Awesome Copilot**：[github.com/github/awesome-copilot](https://github.com/github/awesome-copilot) — Skills/Agents/Instructions汇总
- **Anthropic Skills**：[github.com/anthropics/skills](https://github.com/anthropics/skills/tree/main) — Agent Skills示例与规范

**落地建议**：先把这两个仓库里相关的示例交给AI，让它基于你的项目做筛选，再决定哪些能力值得纳入团队可复用能力库。

## 四、文档体系·Spec Coding与Few-Shot（对应2.8）

### 4.1 为什么不能依赖AI内置文档流程

- 内置流程是通用的，无法覆盖你项目特有的文档规范
- AI内部推理过程中有价值的中间分析默认不会保留
- AI倾向于产出碎片化文档，缺乏全局文档架构意识

### 4.2 维护文件/目录架构的四个目的

1. **全局规划补位**：为AI提供全局上下文，弥补其全局视野不足
2. **过程资产沉淀**：让每次迭代的分析、决策、变更有据可查
3. **缓解上下文窗口有限**：文档是对项目的高密度抽象
4. **问题回溯与设计思路留存**：后续遇到类似问题可快速回溯

### 4.3 Spec Coding vs Vibe Coding

| 概念 | Vibe Coding | Spec Coding |
|------|------------|-------------|
| 做法 | 让AI凭感觉写代码 | 动手前先对齐规格文档 |
| 速度 | 快 | 前期慢，整体快 |
| 典型问题 | 数据结构不兼容、设计决策无据可查 | — |
| 文档产出 | 无 | spec.md + tasks.md + checklist.md |

Spec Coding核心三件套：

| 文档 | 内容 | 作用 |
|------|------|------|
| spec.md | 项目范围：做什么、为什么、边界在哪 | 对齐目标，防止跑偏 |
| tasks.md | 任务拆解：具体步骤、执行顺序 | 明确路径，控制节奏 |
| checklist.md | 验收清单：怎么确认完成 | 可验证，避免"差不多就行" |

> 简单任务直接让AI写就行；复杂或跨模块的任务，先花10分钟对齐规格文档再动手。

### 4.4 Few-Shot参照系

| 参考类型 | 作用 | 典型输入 |
|----------|------|----------|
| 参考代码 | 对齐命名、结构、异常处理习惯 | "按这个Service风格写新模块" |
| 草图/原型 | 对齐页面布局与交互预期 | "按这张草图生成前端页面" |
| 现有文档 | 对齐表达方式与章节结构 | "按该模板生成本次复盘文档" |

> Few-Shot可理解为"低成本规格"：先给样例再给任务，通常比纯文字要求更稳定。

### 4.5 规则沉淀触发器

| 次数 | 动作 |
|------|------|
| 第1次 | 口头说明或临时备注 |
| 第2次 | 标记为高频约束 |
| 第3次 | 正式写入项目记忆/规范文档 |

典型可沉淀项：包管理器统一规则、测试先后顺序、接口前缀、提交规范、目录命名约定。
